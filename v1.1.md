# ng-haptics v1.1 Roadmap

## Objetivo de la versión

La v1.1 NO debe centrarse en añadir muchas nuevas features.

El objetivo principal es:

* consolidar la librería
* mejorar DX
* mejorar estabilidad
* reducir confusión de soporte
* mejorar debugging
* preparar la base para futuras versiones

La prioridad es que `ng-haptics` se sienta:

* sólida
* profesional
* Angular-native
* tiny
* mantenible

---

# Features principales de v1.1

# 1. Support Detection API

## Problema

El soporte de haptics web es inconsistente:

* Android funciona bien
* iOS Safari es limitado
* desktop normalmente no soporta vibración
* algunos navegadores bloquean haptics
* DND/silent mode afectan comportamiento

Actualmente el usuario no tiene una forma clara de saber:

* si el dispositivo soporta haptics
* qué nivel de soporte existe
* por qué no funciona

---

# Objetivo

Añadir una API de detección de soporte.

---

# API esperada

```ts
const support = haptics.support();
```

---

# Resultado esperado

```ts
{
  supported: true,
  platform: 'android',
  method: 'vibration-api',
  reducedMotion: false,
  browser: 'chrome'
}
```

---

# Interfaz esperada

```ts
export interface HapticsSupport {
  supported: boolean;

  platform:
    | 'android'
    | 'ios'
    | 'desktop'
    | 'unknown';

  method:
    | 'vibration-api'
    | 'noop'
    | 'unsupported';

  browser:
    | 'chrome'
    | 'safari'
    | 'firefox'
    | 'edge'
    | 'unknown';

  reducedMotion: boolean;
}
```

---

# Consideraciones técnicas

## Platform detection

Usar:

* `navigator.userAgent`
* `navigator.userAgentData`
* feature detection

Evitar:

* dependencias externas
* librerías UA parser

---

# Browser detection

No buscar exactitud absoluta.
Solo suficiente para:

* debugging
* demo
* feedback visual

---

# SSR Safety

La API debe ser:

* SSR-safe
* browser-only internamente

Nunca acceder a:

* `window`
* `navigator`

fuera de runtime browser.

---

# Demo integration

La demo debe mostrar visualmente:

## Example

```txt
Platform: Android
Browser: Chrome
Support: Full
Reduced Motion: Disabled
```

# 2. Cooldown / Throttling

## Problema

Actualmente es posible hacer spam accidental de vibraciones:

```html
<div (pointermove)="haptics.light()">
```

Esto puede:

* destruir UX
* afectar batería
* generar ruido táctil
* degradar experiencia

---

# Objetivo

Implementar protección interna automática.

---

# Configuración esperada

```ts
provideHaptics({
  cooldown: 40
});
```

---

# Comportamiento esperado

* ignorar vibraciones demasiado frecuentes
* evitar bursts accidentales
* mantener UX suave

---

# Requisitos técnicos

## Debe ser:

* lightweight
* sin timers complejos
* sin RxJS
* sin listeners globales

---

# Implementación recomendada

Usar:

* timestamp interno
* minimum interval

Ejemplo conceptual:

```ts
if (Date.now() - lastTrigger < cooldown) {
  return;
}
```

# 4. Improved Reduced Motion Support

## Objetivo

Mejorar accesibilidad.

---

# Requisitos

Escuchar dinámicamente:

```css
prefers-reduced-motion
```

---

# Comportamiento esperado

Si el usuario activa:

* Reduced Motion

Entonces:

* desactivar haptics automáticamente

Salvo:

```ts
provideHaptics({
  respectReducedMotion: false
});
```

---

# Implementación técnica

Usar:

```ts
window.matchMedia('(prefers-reduced-motion: reduce)')
```

---

# Requisitos importantes

* cleanup listeners correctamente
* SSR-safe
* sin memory leaks
